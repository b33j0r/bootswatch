Getting Started (Mac/Linux)¶

Note

When following these steps, replace mypos with the actual name of your agent integration.

Prerequisites¶

Before we get started, we need to bootstrap our environment with a few tools.

python¶

The Omnivore POS Agent SDK is developed using Python 2.7.9. Python 3 is not supported. To verify that you’re running a compatible python version, type this command at a shell prompt:

$ python --version
Python 2.7.9

If you don’t have it installed, or have another version, please download and install it from the Python 2.7.9 Release Page.

VC++ for Python 2.7 (Windows Only)¶

To install many popular third-party python packages, a C++ compiler is needed. On unix and macOS systems, a compiler is usually available by default. On Windows, a very specific version of the Microsoft Visual C++ compiler is required.

Since this compiler is no longer available commercially, Microsoft has made the Microsoft Visual C++ Compiler for Python 2.7 available for free. You can safely install it along other installations of Visual Studio.

You should install this after installing Python 2.7.9, but before installing any other dependencies.

git¶

You will also need to install git, if you don’t already have it.

Note

If you aren’t familiar with git, consider walking through the quick interactive tutorial, Try Git.

pip¶

pip is the most popular and robust package manager for python. Modern python installers will automatically install it, but you should ensure that you have it:

$ pip --version
pip 8.1.2 from {some-path}/site-packages (python 2.7)

If that command fails (or the indicated python version is not 2.7), you can learn how to install pip here. Once you have pip installed, you should upgrade it, along with a package it uses called setuptools:

pip install -U setuptools pip

Creating a Virtual Environment¶

A virtual environment is a directory which contains a completely independent installation of python, including the standard library, any third-party dependencies that you install, and a collection of tools. They’re great because they don’t modify your system installation of python–when anything goes wrong, you can always delete and recreate a virtual environment.

Note

There are two widely-used approaches to storing virtual environments within your filesystem:

  • Some people put all of them in one directory (e.g. ~/.virtualenvs or C:\virtualenvs).
  • Others put them adjacent to their project code (e.g. ~/myproject/env or C:\myproject\env).

It’s largely a personal preference, so use whatever you’re comfortable with. For the purpose of this guide, we’ll create it near the project directory.

To get started, use pip to install the tool:

pip install -U virtualenv

We’ll create a directory for our projects, then create the virtual environment inside of it. Note that the virtualenv command specifies the path, as well as the name of the virtual environment we’re creating (in this case, env):

mkdir ~/omnivore
virtualenv ~/omnivore/env

To use the newly created environment, we have to “activate” it. This is important enough to repeat: you must activate the virtual environment whenever you start a new shell session, tab, window, etc.

An environment is activated by invoking a script inside its bin directory:

$ source ~/omnivore/env/bin/activate
(env) $

If everything works, your prompt will change to include the name of the virtualenv, as seen in the last line of the example above.

Warning

The remaining steps must be run in a shell in which the virtual environment has been activated. If something doesn’t work, verify that (env) is prepended to your shell prompt.

For simplicity, the rest of the guide will not show the shell prompt.

Getting the Code¶

First, we need to clone the agent-common repository from git (note that we’re cloning it into a from a repository named agent-common-public, but into a directory named agent-common):

cd ~/omnivore
git clone https://github.com/omnivore-io/agent-common-public.git agent-common

agent-common contains the SDK libraries that you will use in your integration code. Next, we’ll clone the repository that the Omnivore SDK team has created for your specific integration.

We’ll provide you with your exact GitHub url, but in the example below you should replace mypos with your POS name:

git clone https://github.com/omnivore-io/agent-mypos.git

This creates a directory ~/omnivore/agent-mypos, where you will write custom code to build your integration.

Installing Dependencies¶

We’ll use pip and setuptools to install the dependencies into the virtualenv:

cd ~/omnivore/agent-common
pip install -r requirements.txt

Finally, we need to tell the virtual environment about our project files. Python uses an environment variable called PYTHONPATH, which is similar to Java’s CLASSPATH. A virtual environment manages PYTHONPATH for you, and keeps it up to date. For local code, such as agent-common and agent-mypos, we use a setup.py script instead of pip.

Note

In documentation for other python projects, you will often see this command invoked as python setup.py install. However, we do not wish to install our local projects. Rather, we want the virtual environment to use our latest code as we work on it. Therefore, we use the develop command.

First, we’ll run setup.py for agent-common:

cd ~/omnivore/agent-common
python setup.py develop

Now do the same thing for your POS-specific project:

cd ~/omnivore/agent-mypos
python setup.py develop

Running the Integration Tests¶

Now you should be able to run the tests:

(omnivore) agent-mypos$ python scripts/test.py --pos-type link

You can also specify -t to change how many tests run. Setting it to -t 1 only runs the tests involved in the first development milestone (basic reads):

(omnivore) agent-mypos$ python scripts/test.py --pos-type link -t 1

See Integration Tests for more information.

The Request Client¶

class RequestClient(config, state, store=None, **kwargs)[source]¶

The primary interface for communicating with the POS system. When creating an integration, this is the most important interface. It ultimately provides all of the functionality such as get_tickets, execute_cash_payment, etc. Unless your integration is trivial, it is recommended that you make use of other objects to actually perform these operations (composition/strategy pattern).

Unlike its long-lived parent object, it is created per-request to handle a specific operation, then it is destroyed. For this reason, transactional resources should be managed by the set_up and tear_down methods.
